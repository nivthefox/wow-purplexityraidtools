1. Overview

PRT Notes lets a raid leader share time-based assignment reminders with the raid. During an encounter, each player sees only the reminders relevant to them—displayed as a static assignment sheet and as timed popups (icons, bars, text, or rings) that fire at the right moment.

What This Feature Is

  • A consumer of externally-authored notes in NSRT's line-based format.
  • A timer engine that fires reminders at the right time based on encounter phase and elapsed time.
  • A display system with four popup styles and a persistent assignment sheet.
  • A communication layer so the raid leader can broadcast notes to the raid.

What This Feature Is Not

  • An in-game note editor. Notes are authored externally and pasted in.
  • A personal note system. All notes are shared raid-wide; there is no per-player note slot.
  • An auto-selection system. PRT never picks a note by detecting the boss; the active note is set by the raid leader—manually or via broadcast.

2. Note Format

PRT adopts NSRT's line-based format. A note is a multiline plain-text string. Each line is either a metadata line, a timed reminder line, or freeform text. Fields within a line are semicolon-delimited key:value pairs. Unrecognized fields are parsed and ignored without error, so notes authored with fields PRT does not support remain loadable.

2.1 Metadata Line

The line containing EncounterID: establishes the encounter context for the note's timed lines. A note contains exactly one metadata line—a note is one encounter's plan. A note containing more than one line with EncounterID: is invalid: it is rejected and not saved—with an error message on save, silently on broadcast receive (see Note Storage & Selection). A note with no metadata line is valid but inert—its lines are all freeform and it never applies to any encounter. 

EncounterID:3176;Name:Sszorak - Mythic;Difficulty:Mythic
FieldRequiredDescription
EncounterIDYesWoW encounter ID (integer). Used to match the note to a live encounter.
NameNoHuman-readable encounter name. For display only.
DifficultyNo"Normal", "Heroic", or "Mythic". When present, the note's reminders only activate on that difficulty; when absent, the note matches any difficulty. Separate Heroic and Mythic plans for the same boss are separate notes.

2.2 Timed Reminder Line

A line is recognized as a timed reminder when it contains all three of: time:, tag:, and at least one of text: or spellid:. It must also appear after the note's metadata line; a timed line before the metadata line is treated as freeform text. 

time:30;tag:Playername1 Playername2;spellid:98008;text:Use Spirit Link;ph:1;dur:8;DisplayType:Icon
FieldRequiredDefaultDescription
timeYes— Seconds into the current phase at which the event happens. The popup appears dur seconds before this moment and counts down to it. Decimal values are accepted.
tagYes— Space-separated list of targets. Determines who sees this reminder. See Tag Matching.
textConditional— Display text. Required if spellid is absent. Supports raid marker tokens ({star}, {circle}, {rt1}–{rt8}).
spellidConditional— Spell ID for icon display. Required if text is absent.
phNo1 Phase number. The timer counts from the start of this phase. Decimal values (e.g. 2.5 for an intermission) are accepted, matching NSRT.
durNo5 Display lead time in seconds. The popup appears this many seconds before the event and disappears when the event occurs. This matches NSRT's behavior exactly.
DisplayTypeNoIcon if spellid present, else Text One of: Text, Bar, Icon, Circle.
bossSpellNo— Boss spell ID. Used for icon display on the static note frame (distinct from the player's spellid).
colorsNo— Space/colon-separated RGBA values. Interpretation depends on DisplayType.
TTSNo— Text-to-speech string, or "true" (uses text field) / "false".
TTSTimerNodur, else 0 Seconds before the event to trigger sound/TTS. Defaults to dur when dur is set (so the alert fires at popup appearance), otherwise 0. Matches NSRT.
countdownNo— If present, announce a numeric countdown starting at this many seconds before the event.
soundNo— Sound to play. A LibSharedMedia sound name or a file path.
glowunitNo— Accepted for NSRT note compatibility; PRT parses and ignores this field. It produces no effect.

2.3 Freeform Lines

Any line that does not match a metadata pattern or a timed reminder pattern is treated as freeform text. Freeform lines are displayed in the static note frame but do not generate timed popups.

3. Note Storage & Selection

3.1 Data Model

Notes are stored in the user's profile under the notes settings key. The structure holds a collection of named notes and tracks which note is currently active. 

profile.notes = {
    enabled = true,
    activeNote = "Sszorak Mythic",       -- name of the currently selected note, or nil
    savedNotes = {
        ["Sszorak Mythic"] = {
            text = "EncounterID:3176;...",  -- the raw multiline note string
        },
        ["Anub'arash Mythic"] = {
            text = "EncounterID:3009;...",
        },
    },
    display = { ... },                    -- display preferences (see Config UI)
    positions = { ... },                  -- frame positions (see Module Integration)
}

3.2 Behaviors

  • Save: When a user pastes text and provides a name, the note is validated and saved under that name. If a note with that name already exists, it is overwritten (with a confirmation prompt).
  • Validation: A note containing more than one metadata line (more than one line with EncounterID:) is invalid. Saving is rejected: the note is not stored, the previous content (if any) is untouched, and the user sees the error "A note may only contain one encounter. Use a separate note per encounter."
  • Delete: A named note can be deleted. If the deleted note was active, activeNote becomes nil.
  • Rename: A named note can be renamed. If it was the active note, activeNote updates to match.
  • Activate: Selecting a note from the dropdown sets it as activeNote and triggers parsing. Only one note is active at a time.
  • Receive: When a note is received via broadcast, it is stored as a named note (using the note name from the broadcast payload, or "Received Note" as fallback) and automatically activated. If a stored note already has that name, the received note overwrites it silently—no confirmation prompt. The broadcaster's copy is authoritative. A received note that fails validation (multiple metadata lines) is rejected silently: not stored, not activated, no message. PRT itself refuses to save such a note, so a valid PRT sender can never broadcast one; this case only arises from hand-crafted or out-of-version payloads. On successful receive, the static note frame is shown immediately—equivalent to pressing the Show/Hide button—so the raid sees the plan the moment it arrives. The receiver can press Hide to dismiss it, and it hides per their "Hide" preference at encounter end as usual.
Note: Notes live inside the profile, so switching PRT profiles switches notes too. This is consistent with how all other PRT settings behave.

4. Parsing

4.1 Parse Trigger

Parsing occurs when:

  • A note is activated (selected from dropdown).
  • A note is received via broadcast.
  • A note's raw text is updated (re-pasted).

4.2 Parse Process

The parser walks the note string line by line and produces a structured representation:

VALIDATE: count lines containing "EncounterID:"
    IF count > 1: the note is invalid (see 3.2 Validation); do not parse

FOREACH line IN note.text split by newline:
    IF line contains "EncounterID:":
        Extract encounterID, name, difficulty
        Set the note's encounter context
        Continue

    IF encounter context is set
       AND line contains "time:" AND "tag:" AND ("text:" OR "spellid:"):
        Extract all recognized fields via pattern matching
        Build a Reminder object:
            reminder.time        = parsed time (number, seconds)
            reminder.tag         = parsed tag string (raw, evaluated later)
            reminder.text        = parsed text or nil
            reminder.spellID     = parsed spell ID or nil
            reminder.phase       = parsed ph or 1
            reminder.duration    = parsed dur or 5
            reminder.displayType = parsed DisplayType or (spellID ? "Icon" : "Text")
            reminder.tts         = parsed TTS or nil
            reminder.ttsTimer    = parsed TTSTimer, or dur when dur was given, else 0
            reminder.countdown   = parsed countdown or nil
            reminder.sound       = parsed sound or nil
            reminder.bossSpell   = parsed bossSpell or nil
            reminder.colors      = parsed colors or nil
        Store reminder under reminder.phase
        Continue

    ELSE:
        Treat as freeform text; store for note frame display

4.3 Parse Output

The parse result describes the note's single encounter:

  • The encounter ID, and metadata (name, difficulty—difficulty may be absent, meaning the note matches any difficulty).
  • A table of reminders keyed by phase number, sorted by time within each phase.
  • A list of freeform text lines.

At encounter start, the active note applies when its encounter ID matches the live encounter and its difficulty either matches the live difficulty or is absent.

5. Tag Matching

A reminder's tag: field is a space-separated list of identifiers. A reminder is relevant to the local player if any identifier in the tag matches the player. Matching is case-insensitive.

Identifier TypeExampleMatches When
Player name Healername1 The local player's name matches (case-insensitive).
Role tank, healer, damager The local player's assigned role matches.
Class (numeric ID) 2 (Paladin) The local player's class ID matches. Numeric identifiers 1–13 are class IDs.
Spec (numeric ID) 65 (Holy Paladin) The local player's specialization ID matches. Numeric identifiers of 62 or higher are spec IDs. (Class IDs and spec IDs occupy disjoint ranges; numeric values 14–61 match nothing.)
Group group1–group8 The local player is in the specified raid subgroup.
Position melee, ranged The local player's role/spec is classified as melee or ranged.
Everyone everyone Always matches.

5.1 When Tag Matching Runs

Tag matching is evaluated once at parse time and again on GROUP_ROSTER_UPDATE (in case the player's group/role changed mid-raid) and PLAYER_SPECIALIZATION_CHANGED (spec-ID and melee/ranged tags depend on the current spec). The result is a filtered list of reminders that are relevant to the local player.

NSRT's nickname matching is not supported. PRT has no nickname system; only actual character names match.

6. Timer Engine

The timer engine is responsible for firing reminders at the correct moment during an encounter. It activates on ENCOUNTER_START and deactivates on ENCOUNTER_END.

6.1 Time References

Every reminder's time: is relative to a reference point. Which reference point depends on the reminder's configuration:

ConfigurationReference Point
ph:1 (or no ph:) Encounter start (phase 1 is synonymous with encounter start).
ph:N (N > 1, decimals allowed) The moment phase N begins, as reported by a boss mod stage callback (see 6.3).

(BigWigs / DBM stage callbacks) PerSecTicker : Per-Second Ticker

AND elapsed >= time

6.2 Encounter Start

When ENCOUNTER_START fires:

  1. Record the encounter ID, difficulty, and start time.
  2. Check whether the active note applies: its encounter ID matches, and its difficulty either matches or is absent (see 4.3). If it does not apply, do nothing.
  3. Set the current phase to 1 and record phase 1's start time as the encounter start time.
  4. Start a per-second ticker that drives timer updates and display refreshes.
  5. Build the static note frame content for this encounter and show the frame, regardless of whether it was previously hidden (see 7.2).

6.3 Phase Tracking

Phase changes come from boss mod stage callbacks. PRT listens for both BigWigs (BigWigs_SetStage) and DBM (DBM_SetStage). These callbacks remain fully functional post-Midnight: the boss mods build their staging on Blizzard's encounter timeline API and report stage numbers as plain Lua numbers. Fractional stages (e.g. 2.5 for intermissions) are reported by the boss mods using the same convention NSRT's notes use, so decimal ph: values map directly.

When a stage callback is received:

  1. Discard the callback if any of its arguments is a secret value (following MRT's guard for boss mod callbacks).
  2. Ignore it if the reported stage equals the current phase—when both BigWigs and DBM are installed, whichever reports a transition first wins and the duplicate is dropped.
  3. Record the new phase number and its start time.
  4. Cancel all unfired reminders from earlier phases and dismiss their popups, matching NSRT: a phase-1 reminder that has not yet fired when phase 2 begins simply never fires.
Dependency: Phase tracking requires BigWigs or DBM to be installed and to have staging support for the encounter. If neither addon is present, only phase 1 reminders fire, and the module logs a warning once at activation.

6.4 Per-Second Tick

Every second while active, the timer engine:

  1. Calculates elapsed time from encounter start.
  2. For each unfired reminder whose tag matches the local player:
    • Determines the reference time (the reminder's phase start).
    • If the reference time is not yet known (the phase hasn't started), skip.
    • Calculates remainingTime = reminder.time - (now - referenceTime)—the seconds until the event.
    • If remainingTime <= reminder.duration and the popup hasn't been shown: show the popup, counting down from remainingTime to zero. This matches NSRT: the popup leads the event by dur seconds and expires when the event occurs.
    • If remainingTime <= reminder.ttsTimer and the audio alert hasn't triggered: fire it once. Sound (sound:) plays if set; TTS fires only when no sound is set (see Ancillary Effects). Both share this single timer, matching NSRT.
    • If countdown is set: announce each whole-second number as remainingTime crosses it within the countdown range, announcing each number at most once. (Times may be decimal, so the announcement fires on crossing the threshold, not only when remainingTime is exactly whole.)
  3. Updates the static note frame's countdown displays.
  4. Updates active popup timers (time remaining until each popup's event).
  5. Removes expired popups (those whose event time has passed).

6.5 Encounter End

When ENCOUNTER_END fires:

  1. Stop the per-second ticker.
  2. Dismiss all active popups.
  3. Handle the static note frame according to the user's "Hide" setting: hide immediately, linger 5 seconds then fade, or ("Never") leave it visible until the active note changes or the user hides it manually.

7. Display: Static Note Frame

The static note frame is a persistent, movable, resizable frame that shows the reminder list for the active encounter. It acts as a living assignment sheet—times count down in real time, and past entries are dimmed, or hidden entirely when the "Hide expired reminders" setting is on.

The frame's contents are governed by the "Show only my assignments" setting (default: on). When on, only tag-matched reminders appear—each raider sees their own sheet. When off, every reminder appears—the view a raid leader uses to review the full plan.

7.1 Content

Each reminder line is rendered as a single row containing:

  • The countdown time in M:SS format, color-coded:
    • > 10 seconds remaining: the frame's normal text color (white/grey).
    • 1–10 seconds remaining: the user-configurable countdown color (default green).
    • Expired: dimmed grey (or hidden, per the "Hide expired reminders" setting).
    • Reference not yet known (the phase hasn't begun): static time in the normal text color, no countdown.
  • Spell icon (from spellid: or bossSpell:), if present.
  • Reminder text, with all player names rendered in their class color, at all times.

Freeform text lines are rendered as-is, without timers.

7.2 Visibility

  • The note frame appears when an encounter starts and the active note applies to that encounter. This is unconditional: it shows even if the frame was previously hidden—by the "Hide" setting at the last encounter's end or by the user pressing Hide. A hide is never sticky across pulls.
  • The note frame appears when a broadcast note is received—equivalent to pressing Show (see Note Storage & Selection).
  • There is no setting that suppresses the note frame while the module is active—the only gates are the module toggle and the content type settings (below), both of which govern the whole module, not the frame alone. A user who never wants the frame disables the Notes module.
  • Re-parsing the active note (e.g. editing the paste box) refreshes the frame's content if the frame is shown. It never changes the frame's visibility—showing and hiding happen only through the triggers in this list.
  • When the encounter ends, the note frame behaves according to the user's "Hide" setting: "Immediately" (default), "Fade" (linger 5 seconds, then fade out), or "Never" (the frame stays visible until the active note changes or the user hides it manually).
  • The Show/Hide button in the config UI toggles the note frame on demand outside an encounter. There is no separate preview mode: the frame renders exactly as it always does, under the same settings ("Show only my assignments", "Hide expired reminders", fonts, colors). With no encounter running, no reference times are known, so every reminder renders per the "reference not yet known" rule in 7.1—static times, no countdowns. The same button is the manual dismissal for the "Never" hide mode.
  • The note frame is only shown during content types the user has enabled (using PRT's existing content type system).

7.3 Frame Properties

  • Movable and resizable (position and size saved to the profile, following CooldownRoster's pattern).
  • Lockable (prevents accidental dragging, following CooldownRoster's pattern). Locked by default.
  • Scrollable if content exceeds frame height.
  • Configurable font face, size, and outline; configurable background color and opacity.

8. Display: Active Reminder Popups

A reminder's popup appears dur seconds before its event, counts down to zero at the moment of the event, then fades out. Multiple popups stack vertically. The "Enable popups" master toggle (see Configuration UI) disables the entire popup system; when off, no popup of any type ever appears, regardless of note content.

8.1 Display Types

TypeVisualBest For
Icon Square spell icon with a cooldown swipe animation, text label beside it, and a timer. Player ability assignments ("Use Aura Mastery").
Bar Horizontal status bar with spell icon, text label, and timer. Bar drains over the countdown, emptying at the event. Duration-based events ("Dodge Breath for 8 seconds").
Text Plain text with an optional countdown in parentheses. Simple callouts ("Spread Now").
Circle Circular ring with a cooldown-style sweep, text label in center, timer. Rotational cooldowns, alternative visual preference.

8.2 Popup Arrangement

  • Each display type has its own anchor point (mover frame), independently positionable.
  • Multiple simultaneous popups of the same type stack in a configurable direction (up or down).
  • Stacked popups are sorted by expiration time (soonest-to-expire on top).
  • Mover frame positions are saved to the profile.

8.3 Popup Lifecycle

WHEN remainingTime until the event <= dur:
    Create popup frame for the reminder's DisplayType
    Set icon from spellid (if present)
    Set text from text field
    Set countdown to remainingTime (counts down to the event)
    Apply colors from colors field (if present)
    Start cooldown/swipe animation over the countdown
    Arrange all active popups of this type
    (Sound/TTS is NOT triggered here; it fires on the shared
     TTSTimer timer—see 6.4 and 9.2.1)

EVERY FRAME (OnUpdate):
    Update countdown display
    Update swipe/bar progress

WHEN the event time arrives (countdown reaches zero):
    Fade out popup (0.5 second fade)
    Release the frame
    Re-arrange remaining popups

9. Display: Ancillary Effects

9.1 Text-to-Speech

If TTS is present and not "false", the addon speaks the TTS string (or the reminder's text if TTS is "true") using WoW's built-in C_VoiceChat.SpeakText.

9.2 Sound

If sound is present, the addon plays the named sound via LibSharedMedia (or as a direct file path).

9.2.1 Audio Alert Timing

Sound and TTS share a single timer and are mutually exclusive, matching NSRT: the alert fires once, TTSTimer seconds before the event. If sound is set, the sound plays and TTS is skipped; TTS fires only when no sound is set (or the sound fails to resolve). Because TTSTimer defaults to dur, the common case is an alert at the moment the popup appears; an explicit TTSTimer moves it. Both dur and TTSTimer are clamped to the reminder's time.

9.3 Countdown

If countdown is present, the addon announces decreasing numbers (via TTS) starting at countdown seconds before the event. For example, countdown:3 announces "3", "2", "1". Countdown announcements are independent of the "Enable TTS" toggle: they play even when TTS is disabled.

9.4 Unit Frame Glow

PRT does not implement unit frame glow. The glowunit field is parsed for format compatibility and discarded. It produces no visual effect, no error, and no warning.

10. Communication

PRT has no existing communication infrastructure. This feature introduces addon messaging for the first time.

10.1 Import / Export

  • Import: The user pastes a raw NSRT-format note string into the config UI's text box. The raw string is the import format—no encoding, no compression.
  • Export: The user copies the raw note string from the text box. Same format as import.

10.2 Broadcast

The raid leader (or assistant) can broadcast the active note to the raid. All raid members running PRT receive it, store it, and activate it.

10.2.1 Who Can Broadcast

Only players who are the raid leader or have raid assistant can broadcast. For all other players, the "Send" button is visible but disabled, with a tooltip reading "Requires raid leader or assistant."

Broadcasting during combat is not allowed, ever. The Send and Clear buttons disable while the local player is in combat, with a tooltip reading "Cannot send during combat"—and the restriction is enforced in the send path itself, not merely the button state: a send invoked while in combat is refused, so no code path (present or future) can broadcast mid-combat. This is a product decision, not a technical limit: mid-pull note swaps are not supported. Consequently, receivers never handle a note arriving mid-combat from a compliant sender, and show-on-receive (see 3.2) needs no combat special-casing.

10.2.2 Broadcast Behavior

WHEN raid leader clicks "Send":
    Serialize the active note's raw text string (LibSerialize)
    Compress the serialized payload (LibDeflate)
    Send to the RAID channel via AceComm, which chunks to fit
        WoW's addon message size limit
    Include the note name in the message payload

WHEN a raid member receives a complete note:
    IF sender is raid leader or assistant:
        Decompress and deserialize the payload
        IF the note fails validation (multiple metadata lines):
            Discard silently and STOP (see 3.2 Receive)
        Store as a named note (using the note name from the payload)
        Activate it (set as activeNote, trigger parse)
        Show the static note frame (equivalent to pressing Show)
        Display a chat notification: "Received note: [NoteName] from [Sender]"
    ELSE:
        Ignore the message

10.2.3 Clear

The raid leader can broadcast a "clear" message that deactivates the current note on all raid members. This does not delete stored notes—it only sets activeNote to nil. Receivers apply the same sender check as note broadcasts: clear messages from anyone who is not the raid leader or an assistant are ignored.

New infrastructure: The communication layer (serialize, compress, send/receive, leader-check) is new to PRT. It is built as a shared utility so future features can reuse it, not baked into the Notes module.

11. Configuration UI

Notes registers a tab in PRT's config frame. The tab is organized into sections.

11.1 Note Management Section

  • Note selector dropdown: Lists all saved note names plus a (none) entry. Selecting a name activates that note; selecting (none) deactivates the current note locally.
  • New Note button: Prompts for a name, then creates an empty note and opens it for pasting.
  • Delete button: Deletes the currently selected note (with confirmation).
  • Rename button: Renames the currently selected note.
  • Paste box: A large multiline text area showing the active note's raw text. Editable—pasting new content overwrites the note and triggers a re-parse. This is the primary input mechanism. A re-parse refreshes the note frame's content if the frame is currently shown; it never changes the frame's visibility (see Static Note Frame).
  • Send button: Broadcasts the active note to the raid. Always visible; enabled only when the player is raid leader or assistant AND a note is active AND the player is out of combat. When disabled due to permissions, the tooltip reads "Requires raid leader or assistant"; when disabled due to combat, "Cannot send during combat."
  • Clear button: Broadcasts the clear message (see Communication), deactivating the current note raid-wide. Gated identically to Send: always visible, enabled only for the raid leader or an assistant while out of combat, with the same tooltips when disabled.

12. Module Integration

12.1 Module Registration

Notes follows PRT's standard module pattern:

  • Registers via PRT:RegisterModule("notes", NotesModule).
  • Declares defaults via PRT.defaults.notes.
  • Registers a config tab via PRT:RegisterTab("Notes", ...).

12.2 Lifecycle

HookBehavior
Initialize() Create display frames (note frame, popup mover frames). Set up the communication channel.
IsActivatable() Not implemented. PRT treats an absent hook as always-activatable (the AutoInvite precedent), which is the intent: the module works solo—note management, the Show/Hide button, and the paste box do not require a group. Group-dependent behaviors (broadcast, encounter activation) gate themselves.
OnEnable() Register for ENCOUNTER_START, ENCOUNTER_END, GROUP_ROSTER_UPDATE, PLAYER_SPECIALIZATION_CHANGED, and the BigWigs/DBM stage callbacks. Parse the active note. Begin listening for broadcast messages.
OnDisable() Unregister events. Stop the timer engine. Dismiss popups. Hide frames.

12.3 Content Types

Notes uses PRT's existing content type system. The module's contentTypes defaults enable Raid (all difficulties) and disable other content types, since timed boss notes are a raid-centric feature.

12.4 Position Persistence

Frame positions (note frame + each popup type's mover) are stored under profile.notes.positions, following CooldownRoster's pattern.
